# ライブ提出(WebRTC)+ 名簿ランチャー 設計書

日付: 2026-07-19
対象: スキーマ療法カード(単一 `index.html`)。既存のリンク提出機能([[2026-07-19-workshop-submission-design]])を土台に、ワークショップ中の「名前を言う→クリック→机が即表示」をシームレスにする。

## 背景と狙い

リモート(Teams)ワークショップで、講師が「山田さんのを見ましょう」と言った瞬間にその人の机を出したい。現行のリンク方式は「参加者がリンクをコピーしてTeamsに貼る→講師が開く」の送信手間が残る。これを **WebRTCで直送(コピー不要)** にし、届いた提出を **名簿から名前で即呼び出す** UIに刷新する。社内/在宅回線でP2Pが通らない場合に備え、**リンク方式へ自動フォールバック** する。

## 全体像

- **トランスポート層**: ライブ(WebRTC/PeerJS)を第一手段、リンク(既存)をフォールバック。両者は **同一のペイロード**(既存 `v:1` 提出データ)を運ぶ。
- **UI層(名簿ランチャー)**: 提出の届き方に依らず共通。講師モード=名前チップ一覧、机ビューアに名前スイッチャー常駐。

## トランスポート: ライブ提出(WebRTC / PeerJS)

### ライブラリ読込
- PeerJS を **リポジトリ同梱**(`peerjs-1.5.4.min.js`、同一オリジン)。外部CDNの供給網リスク(汚染・MITM)と当日のCDN到達性リスクを避けるため、CDNではなく同梱を採用(コードレビュー M-2 対応)。ライブ機能を使う瞬間(講師の「ライブ提出を開始」、参加者が `#join=` で入る/提出時)にのみ `loadPeerJS()` で1回だけ動的 `<script>` 挿入。
- 読込失敗時は `window.Peer` が無いので、その経路はリンク方式にフォールバック(失敗はキャッシュせず次回再試行= M-1 対応)。基本の自己ワークは完全オフラインのまま(PeerJSに触れない)。
- なお WebRTC の signaling は PeerJS 既定ブローカーを経由する(ペイロード本体はP2P)。同梱はスクリプト配信経路のリスクを断つもので、signaling依存はWebRTCの性質上残る。

### ルームコードと接続ID
- 講師がライブ開始時にコード生成: 5文字の base32(視認性のため `I O 0 1` を除く英数、例 `K7Q2F`)。
- PeerJS の peer id は名前空間付き `stcard-live-<code>`。`unavailable-id`(衝突)時はコードを再生成してリトライ(最大数回)。
- 参加リンク: `https://alcatia9500.github.io/schema-therapy-card/#join=<code>`。QRも表示(講師の共有用。QRは軽量な内蔵SVG生成 or 外部画像に頼らず、まずはコード+リンクのみで実装し、QRは任意拡張)。

### 講師側フロー
1. 講師モードで「ライブ提出を開始」→ `loadPeerJS()` → `new Peer("stcard-live-"+code)`。
2. `peer.on("connection", conn)` → `conn.on("data", payload)` で提出を受信。
3. 受信ペイロードを検証(`decodeSubmission` と同じ形状チェックを共有する `validatePayload()`)→ `addSubmission`(重複は名前+`submittedAt`で排除)→ 名簿を再描画 →「〇〇さんが提出しました」トースト。
4. `conn.on("open")` で受領ACK(`{ok:true}`)を返し、参加者側の成功判定に使う。
5. コード・参加リンクを画面表示。講師はTeamsに1回貼る。タブは開いたまま(閉じると受付停止=バナー表示)。

### 参加者側フロー
- 起動時 `#join=<code>` を検出したら、`state.live = { code, role:"participant" }` を持ち、コードを `sessionStorage`(`stc.liveCode`)に保持。通常どおりワーク開始(自己保存は従来どおり)。
- 「提出」ダイアログで名前・メモ入力後「提出」:
  - ライブコードがある場合: `loadPeerJS()` → `peer.connect("stcard-live-"+code)` → `open` で `conn.send(payload)` → ACK受信で成功トースト「提出しました」。
  - **タイムアウト(8秒)/接続失敗/読込失敗** → 自動でリンク生成UI(既存 `showSubmitResult`)に切替。「うまく送れなかったのでリンクを講師に送ってください」と明示。
  - ライブコードが無い(直接アクセス)場合: 従来のリンク生成のみ。
- 参加者の PeerJS id はランダム(自動)。

### プライバシー / 制約
- ペイロード本体はP2Pデータチャネルのみを流れる。signalingメタ+ルームコードはPeerJSブローカー経由。第三者DBには保存しない。
- 講師受信分も講師端末の localStorage(`schemaTherapyCard.inbox.v1`)へ保存し再読込後も残る。
- 依存が「フォント」に加え「PeerJS(CDN, ライブ時のみ)」も増える。

## UI: 名簿ランチャー(全方式共通)

### 講師モード(受信箱の刷新)
- ライブ状態パネル: 未開始なら「ライブ提出を開始」。開始後はコード・参加リンク・接続数・「コピー」。
- **名前チップ一覧**: 受信済み提出を名前ボタン群で表示。クリックで即 `viewer`。メモありは小バッジ。
- **絞り込み検索窓**: 入力で名前部分一致フィルタ(大人数対応)。
- 既存の手貼り付け欄(複数リンク一括対応)・JSON書き出し/読み込み・個別削除は維持。

### 机ビューア(閲覧専用)にスイッチャー常駐
- ツールバーに **名前ドロップダウン**(受信順/名前順)+ **◀ ▶** ボタン。选択・押下で `viewingId` を切替え再描画。
- **キーボード**: `←`/`→` で前後の提出へ、`Esc` で名簿へ戻る。入力欄フォーカス中は無効。
- 既存の視点操作(ズーム/傾き/パン、閲覧専用でドラッグ不可)は維持。

### リンクを開いた場合
- `#s1=`/`#s0=` を検出したら、確認ダイアログと一覧を挟まず: デコード→ `addSubmission`(重複排除)→ `viewingId` セット→ 即 `viewer`。上部に「保存済み ／ ← 名簿」の細いバー。デコード失敗時のみバナーで案内(既存)。
- `#join=<code>` は参加者ライブ経路(上記)。

## 共有ロジックの整理

- `validatePayload(obj)`: `v===1 && Array.isArray(selectedCardIds) && placements` を判定。`decodeSubmission` とライブ受信の両方が使う(重複コード排除)。
- `dedupKey(payload) = name + "|" + submittedAt`。`addSubmission` は同キー既存なら追加せず既存を返す(直行ビューで既存を開く)。
- ライブ/リンク/一括貼り付け/インポートの全入口が `validatePayload`→`addSubmission` を通る。

## エラー処理

- PeerJS読込失敗: ライブ経路のみ無効化しリンクへ。講師側は「ライブを開始できませんでした。リンク方式をご利用ください」バナー。
- 講師未起動/コード誤り: 参加者接続失敗→リンクフォールバック。
- 講師タブが受付停止(peerエラー/切断): 講師画面に再開ボタン+バナー。
- 既存の localStorage 不可・保存失敗・破損は従来のバナー機構を流用。

## 実装方針

- 既存 `index.html` に追記。ライブ関連は `state.live` と数関数(`loadPeerJS`/`startLiveHost`/`sendLive`)に隔離し、**全経路をtry/catchでガード**。ライブが何で失敗しても基本アプリとリンク方式は壊れない。
- 画面ルーティングは既存の `home/select/canvas/inbox/viewer` を流用(新画面は増やさない)。ライブUIは `inbox` 画面内のパネルとして実装。

## 検証

- **ローカルで可能**: `node --check`;DOMスタブで 名簿描画・名前フィルタ・ビューア名前スイッチャー(dropdown/◀▶/キー)・リンク直行→viewer・複数リンク一括取込・重複排除;リンク往復(既存, 変更なし)。ペイロードのwire形式はライブ/リンクで同一であることを確認。
- **サンドボックスで不可(要ユーザー)**: 別回線2端末での実機WebRTC疎通。→ **本番前に講師役+参加者役の2端末ドライラン必須**。フォールバックにより最悪でもリンク方式で成立。
- headless Chrome で 講師モード(名簿・ライブ開始前パネル)と ビューア(スイッチャー)の描画確認。PeerJSのCDN到達はサンドボックスで不可の可能性があり、その場合はガードされたフォールバック動作(リンクUIが出る)を確認する。
