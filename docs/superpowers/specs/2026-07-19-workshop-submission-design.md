# ワークショップ提出機能 設計書

日付: 2026-07-19
対象: スキーマ療法カード(単一 `index.html`、依存ゼロ、GitHub Pages 公開)

## 目的

ワークショップで参加者が各自の端末で行ったワーク(選んだカード・机の配置・気づきメモ)を講師に提出し、講師がその場でも後日でも一覧から選んで机を再現・確認できるようにする。

## 方式の決定

**リンク提出方式**を採用する。提出データはURLフラグメントに埋め込み、サーバーを使わない。

- 検討した代替案: WebRTC直接接続(理想UXだが社内ネットワークで接続失敗リスク・依存増・実装大)、クラウドDB(非同期提出可だが心理ワークの個人データが第三者サーバーに残る・設定運用負担)。いずれも見送り。
- 採用理由: サーバー不要でどのネットワークでも確実に動き、個人データがリンクの中と講師のブラウザ内にしか存在しない。依存ゼロ・静的ページという既存方針を維持できる。
- トレードオフ: 提出の「送信」はTeams/メール等への手動貼り付け1ステップが残る。講師側はリンクをクリックするだけで自動登録されるため、講師の手間は増えない。

## 提出データとリンク形式

### ペイロード

```json
{
  "v": 1,
  "name": "山田",
  "note": "気づきメモ(任意、最大500文字)",
  "submittedAt": "2026-07-19T05:00:00.000Z",
  "selectedCardIds": ["abandonment", "..."],
  "placements": { "abandonment": { "…既存実装の配置フィールドをそのまま流用…": 0 } },
  "view": { "zoom": 1, "tilt": 30, "panX": 0, "panY": 0 }
}
```

- `selectedCardIds` / `placements` / `view` は既存セッション構造をそのまま流用する。
- `name` は必須(最大40文字)、`note` は任意(最大500文字)。

### エンコード

1. JSON文字列化 → UTF-8バイト列。
2. `CompressionStream("deflate-raw")` で圧縮 → base64url化 → `#s1=<データ>`。
3. `CompressionStream` 非対応ブラウザでは非圧縮のbase64urlに自動フォールバック → `#s0=<データ>`。
4. 完成形: `https://alcatia9500.github.io/schema-therapy-card/#s1=...`

デコード側は接頭辞(`s1`=deflate-raw, `s0`=非圧縮)で分岐する。`DecompressionStream` 非対応の講師ブラウザで `s1` リンクを開いた場合は「新しいブラウザで開いてください」と案内する(現実的にはモダンブラウザ前提で問題にならない)。

- URLフラグメント(`#`以降)はHTTPリクエストに含まれずGitHub Pagesのログに残らない。
- エンコード/デコードは async 関数(Streams API使用)として実装する。

## 参加者側

- 机(canvas)画面のツールバーに「**講師に提出**」ボタンを追加。
- 押下でダイアログ表示:
  - 名前(必須、テキスト入力)
  - 気づきメモ(任意、textarea、「このワークで気づいたこと」プレースホルダ、500文字カウンタ)
  - 「提出リンクを作成」ボタン
- リンク生成後:
  - `navigator.share` が使える環境(主にスマホ)では共有シートを開くボタンを主表示。
  - 併せて「リンクをコピー」ボタン(`navigator.clipboard.writeText`、失敗時はテキストボックス表示+手動コピー案内)。
  - 「このリンクを Teams やメールで講師に送ってください」と明記。
- 提出しても参加者自身のセッション(localStorage)には影響しない。名前・メモはセッションには保存せず、ダイアログ入力の都度使い切り(再提出時に名前だけ前回値を `sessionStorage` に記憶して補完)。

## 講師側(講師モード)

### 受信箱ストレージ

- キー: `schemaTherapyCard.inbox.v1`(既存の `schemaTherapyCard.v1` とは別領域。講師自身のワークと干渉しない)。
- 形式: `{ version: 1, submissions: [ { id, receivedAt, payload } ] }`。`id` は既存 `genId()` を流用。

### 提出リンクの受信

- ページ読み込み時と `hashchange` 時に `#s1=` / `#s0=` を検知したらデコードし、確認画面「〇〇さんの提出を受け取りました。受信箱に保存しますか?」→ 保存で受信箱に追加し講師モードの一覧へ遷移。フラグメントは `history.replaceState` で即座にURLから消す(履歴・ブックマークに個人データを残しにくくする)。
- 同一人物の再提出は上書きせず両方保持(提出時刻で区別)。
- 壊れたリンク(デコード失敗)は「リンクが不完全です。送信者にもう一度コピーしてもらってください」と表示。

### 講師モード画面

- 入口: ホーム画面下部に控えめなリンク「講師モード」。
- 一覧: 名前・提出時刻・カード枚数・メモ有無を行表示。行選択で閲覧画面へ。
- 補助機能:
  - リンク手貼り付け欄(クリックで開けない環境のフォールバック。貼り付け→受信箱に追加)
  - 提出の個別削除(確認ダイアログあり)
  - 全提出のJSONファイル書き出し(`Blob`+`a[download]`)/読み込み(`input[type=file]`)。ブラウザ移行・バックアップ用。

### 閲覧画面(机の再現)

- 既存の canvas 描画を**閲覧専用モード**で流用: ズーム・チルト・パンは操作可、カードのドラッグ移動は不可(参加者の配置を保全)。
- 画面端に提出情報パネル: 名前・提出時刻・気づきメモ。
- 閲覧してもデータは変更されない(受信箱は読み取りのみ)。
- 「一覧に戻る」で講師モード一覧へ。

## エラー処理

- localStorage 不可・保存失敗: 既存の警告バナー機構(`warningBanner()`)に受信箱系のメッセージを追加して乗せる。
- メモ500文字・名前40文字超過: 入力時にカウンタと超過警告、生成ボタン無効化。
- 受信箱JSONの破損: 既存 `loadDB()` と同様にバックアップ退避+初期化+バナー通知。

## 実装方針

- 既存の単一 `index.html` に追記する(依存ゼロ・ビルド不要を維持)。
- 画面は既存の `state.screen` ルーティング(`home`/`select`/`canvas`)に `inbox`(講師一覧)と `viewer`(閲覧)を追加。
- canvas 描画の閲覧専用化は、既存 `renderCanvas` にreadonlyフラグを渡す形で共通化する(コピー実装はしない)。

## 検証

既存の流儀を踏襲する(自動テスト基盤なし):

- `node --check` 相当のJS構文チェック。
- DOMスタブでのロジックテスト: エンコード→デコードの往復一致(圧縮/非圧縮両系)、受信箱の追加・削除・重複提出の共存、名前/メモの長さ制限。
- headless Chrome(`--dump-dom`/`--screenshot`)で参加者側ダイアログ・講師モード一覧・閲覧画面の表示確認。
- 実URL長の確認: 18枚全選択+メモ500文字でリンクが実用長(目安2,000文字以内)に収まること。
